# Documentation Final Audit

<!-- Версия: 1.3 | Обновлено: 2026-04-28 | Платформа: BPMSoft 1.9 -->
<!-- Теги: audit, documentation, navigation, quality, encyclopedia -->

> Итоговая ревизия документации после серии platform dives. Документ фиксирует
> текущее покрытие, правила навигации и критерии добавления новых страниц.
> **Замыкание** как единая модель знаний описано в [`ENCYCLOPEDIA.md`](ENCYCLOPEDIA.md).

## Текущее состояние

Документация покрывает основные практические контуры платформы:

- core architecture, naming, packages, configuration and build;
- Entity Schema, ESQ, data access beyond ESQ;
- WCF/REST services, client modules, NUI, Process, Quartz;
- Security, licensing, operations, SSP/portal;
- Activity/email, mailbox/Exchange/calendar sync, Email Mining;
- reports, files, import, deduplication, mobile, analytics;
- admin/configuration, features, SysSettings, System Designer;
- Global Search, CTI, KB/cases, ESN, notifications;
- OCC/Sender, Integration Tools, OAuth20Integration, AI/ML.

После capstone-пакета документация считается достаточной для регулярной
разработки: задачу можно искать по `INDEX.md`, агенту — по `AGENT_INDEX.md`;
конституция и проверки замыкания — по `ENCYCLOPEDIA.md` и `VERIFY.md`;
быстрые workspace-маршруты лежат в `.cursor/rules/bpmsoft-docs.mdc`.

## Primary entry policy

Для новых задач открывайте overview/dive-страницу, а не legacy-краткий файл:

| Тема | Primary entry |
| ---- | ------------- |
| Конституция энциклопедии (замыкание, домены, процедуры) | `docs/ENCYCLOPEDIA.md` |
| Корень приложения конфигурации, Autogenerated у корня workspace | `docs/architecture/configuration-workspace-layout.md` |
| Entity Schema | `docs/server/entity-schema-overview.md` |
| EventListeners | `docs/server/event-listeners-overview.md` |
| Services | `docs/server/services-overview.md` |
| Process | `docs/server/process-overview.md` |
| File Storage | `docs/server/file-overview.md` |
| Client UI | `docs/client/client-module-overview.md` |
| Admin/configuration | `docs/server/admin-configuration-overview.md` |
| Security | `docs/server/security-overview.md` |
| Runtime diagnostics | `docs/server/diagnostics-deployment-logging-overview.md` |
| Хостинг / деплой / контуры вне `Autogenerated/Src` | `docs/architecture/infrastructure-deployment-boundaries-overview.md` |

Legacy-файлы вроде `entity-schemas.md`, `services.md`, `processes.md` и
`file-storage.md` остаются справочными слоями. Они не должны конкурировать с
overview как основной вход.

## Navigation ownership

| Файл | Роль |
| ---- | ---- |
| `docs/ENCYCLOPEDIA.md` | конституция замкнутой энциклопедии: аксиомы, primary entry по доменам, процедуры |
| `docs/VERIFY.md` | обязательные проверки (ссылки, сироты, опционально lint и регенерация имён схем) |
| `docs/INDEX.md` | человекочитаемое оглавление, learning path, task routing |
| `docs/AGENT_INDEX.md` | быстрый routing для агента по файлам, задачам и символам |
| `.cursor/rules/bpmsoft-docs.mdc` | короткая подсказка для работы с `Autogenerated/Src` |
| `docs/DOCUMENTATION_FINAL_AUDIT.md` | состояние покрытия и правила дальнейшего расширения |

Если добавляется новая крупная страница, обновляйте минимум `INDEX.md` и
`AGENT_INDEX.md`. Если страница должна помогать при работе с кодом, обновляйте
также `.cursor/rules/bpmsoft-docs.mdc`. Перед merge запускайте `docs/tools/Verify-Docs.ps1` (см. `VERIFY.md`).

## Known gaps

| Gap | Решение |
| --- | ------- |
| `INDEX.md` большой; дубли оглавления | логический корень доменов и primary entry — [`ENCYCLOPEDIA.md`](ENCYCLOPEDIA.md); `INDEX` остаётся полным оглавлением |
| `entity-catalog.md` описателен построчно; отставание от кода | машинный сверочный список: `reference/entity-schema-names.generated.json`; команда в [`VERIFY.md`](VERIFY.md) |
| Mailbox / Activity / Email Mining пересекаются | как раньше: `mailbox-exchange-sync-overview.md` как runtime entry |
| Несколько входов в OAuth | как в primary entry table: `auth-oauth20-integration-overview.md`, `auth-sso-ldap-overview.md` |
| Инфраструктура узла вне кода | boundary: `architecture/infrastructure-deployment-boundaries-overview.md` |

Закрытые ранее технические долги (реализация см. [`VERIFY.md`](VERIFY.md)):

- обязательная проверка внутренних ссылок и «сирот» — [`tools/Verify-Docs.ps1`](tools/Verify-Docs.ps1);
- единые правила и команда markdownlint для `docs/` — [`package.json`](package.json) и [`.markdownlint.json`](.markdownlint.json).

CI в репозитории не добавляется автоматически: при желании включите команду из `VERIFY.md` в ваш pipeline сборки решения.

## When to add new docs

Добавляйте новую страницу, если выполняется хотя бы одно условие:

- появляется самостоятельная подсистема с отдельной моделью данных и runtime;
- тема повторяется в нескольких доменных документах и требует boundary-page;
- есть типовая ошибка, которую невозможно хорошо описать в существующем
  troubleshooting;
- существующая страница становится слишком длинной и смешивает разные задачи;
- новый документ улучшает task routing в `INDEX.md` и `AGENT_INDEX.md`.

Не добавляйте страницу, если это один частный пример, который лучше поместить в
pattern catalog или troubleshooting существующего dive.

## Verification baseline

Обязательно для любого merge, затрагивающего `docs/**/*.md`:

```text
powershell -NoProfile -ExecutionPolicy Bypass -File docs/tools/Verify-Docs.ps1
```

Дополнительно (перед значимым релизом набора доков):

```text
cd docs && npm install && npm run lint
powershell -ExecutionPolicy Bypass -File tools/Export-EntitySchemaNames.ps1
```

При необходимости: ReadLints по изменённым файлам в редакторе.

Старшие markdownlint-долги в отдельных файлах устраняются постепенно; новые страницы
должны проходить `npm run lint` или явно указанное исключение в задаче на ревью.
