## Общая модель

Бизнес-процесс состоит из metadata-схемы и исполняемого runtime-класса.

```text
ProcessSchema
  ├── Parameters
  ├── FlowElements
  └── sequence / gateway / task metadata

Process
  ├── runtime state
  ├── Get<T>() / Set<T>()
  └── ProcessModel / ScriptTask methods
```

В autogenerated code часто видна связка:

- `{ProcessName}Schema : ProcessSchema`;
- `{ProcessName} : Process`;
- `{ProcessName}MethodsWrapper : ProcessModel`;
- вложенные `*FlowElement` классы.

## Когда использовать процесс

Процесс уместен, если нужно:

- описать бизнес-сценарий с шагами и условиями;
- дать настройку в Process Designer;
- связать пользовательские действия, записи и ожидание событий;
- запускать сценарий вручную, из сервиса или по расписанию;
- логировать выполнение в системных журналах процессов.

## Когда процесс не лучший выбор

Не стоит переносить в процесс:

- простую синхронную операцию одного сервиса;
- короткую техническую функцию без бизнес-маршрута;
- тяжёлую batch-обработку без Quartz/job-обвязки;
- код, который проще сопровождать как C# service/helper.
