> Каталог рабочих паттернов ML / Prediction / Scoring из `Autogenerated/Src`.

## ProblemType as binding name

**Когда использовать:** нужно выбрать predictor по типу ML-задачи.

**Паттерн:**

```text
MLModel.MLProblemTypeId
  -> problemType.ToString().ToUpper()
  -> ClassFactory.Get<IMLEntityPredictor>(name)
  -> ClassFactory.Get<IMLBatchPredictor>(name)
```

Новый problem type должен иметь binding для single и/или batch prediction.

## External ML service proxy

**Когда использовать:** ML runtime вызывает внешний сервис обучения или
предсказаний.

`MLServiceProxy` инкапсулирует:

- service URL;
- API key;
- RestSharp client;
- endpoint names;
- timeout-ы;
- преобразование HTTP-ошибок в platform exceptions/notifications.

Не размазывайте внешние ML calls по user tasks или UI-facing services.

## Self-rescheduling ML jobs

**Когда использовать:** job должна работать периодически, но регистрируется как
immediate job при старте.

`MLBatchPredictionJob` и `MLModelTrainerJob`:

- читают период из syssetting;
- выполняют работу;
- в `finally` вызывают `SchedulerUtils.ScheduleNextRun`;
- не перепланируются, если период равен `0`.

## Model loader before runtime action

**Когда использовать:** перед prediction/training нужна проверка готовности.

Runtime не работает напрямую с entity `MLModel`. Он загружает `MLModelConfig`
через loader, который проверяет состояние, instance id, service URL, root schema
и настройки колонок.

## Prediction saver as boundary

**Когда использовать:** результат нужно сохранить в target entity и/или
`MLPrediction`.

Predictor возвращает доменный result, а `MLPredictionSaver` отвечает за запись:

- predicted field в entity;
- `MLPrediction`;
- score/classification-specific format.

## Process filter conversion

**Когда использовать:** process user task передаёт фильтры в batch prediction.

`MLDataPredictionUserTask.FilterEditConverter` переводит process filters в NUI
filters и разрешает process parameters перед вызовом `IMLBatchPredictionJob`.

## BaseScoring service facade

**Когда использовать:** внешнему scoring engine нужен scoring map или sync data.

`ScoringService` остаётся тонкой WCF-обёрткой, а логика живёт в
`ScoringEngine`. Методы, доступные извне, проходят auth key validation.

## MLScoring as bridge

**Когда использовать:** scoring должен работать как ML problem type.

`MLScoringEntityPredictor` и `MLBatchScorer` регистрируются с binding name
`MLConsts.ScoringProblemType`, но используют общую ML infrastructure.

## Чего не делать

- Не смешивать `MLModel` и `LlmModel`.
- Не вызывать внешний ML API из client-side JS.
- Не обходить `MLModelLoader` перед prediction.
- Не добавлять новый problem type без predictor binding.
- Не логировать API keys и request bodies с чувствительными данными.
