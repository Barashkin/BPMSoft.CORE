# OCC Outgoing Connector Flow

<!-- Версия: 1.0 | Обновлено: 2026-04-27 | Платформа: BPMSoft 1.9 -->
<!-- Теги: OCC, connector, outgoing, SendMessage, status callback -->

> Как OCC отправляет сообщения из BPMSoft во внешний connector, как
> сохраняются внешние идентификаторы и как статусы возвращаются обратно в
> операторский UI.

## Область

Этот документ дополняет входящий pipeline из
[occ-request-pipeline.md](occ-request-pipeline.md). Здесь фокус на обратном
направлении:

```text
Operator UI / API
  -> BPMSoftOCCChatService.SendMessage
  -> ApiService.SendMessage
  -> Api.SendMessage
  -> external connector
  -> BPMSoftOCCChatRequestService.SendMessageStatus*
  -> BPMSoftOCCChatMessageStatusRequestHandler
  -> MsgChannel "BPMSoftOCCChat"
```

## Серверные точки входа

| Слой | Роль |
| ---- | ---- |
| `BPMSoftOCCChatService.SendMessage(...)` | WCF-метод operator UI |
| `ApiService.SendMessage(...)` | создаёт/обновляет OCC message и делегирует отправку |
| `Api.SendMessage(...)` | формирует connector request и обрабатывает response |
| `BPMSoftOCCChatMessageStatusRequestHandler` | обновляет статусы по callback'ам |
| `BPMSoftOCCChatMessageEditMessageRequestHandler` | обрабатывает результат edit/delete |

Опорный файл исходящей отправки:
`Autogenerated/Src/BPMSoftOCCApi.BPMSoftOCC.cs`.

## Что сохраняется в OCC

Для отправленного сообщения важны поля сущности `BPMSoftOCCChatMessage`:

| Поле | Назначение |
| ---- | ---------- |
| `ExternalMessageId` | id сообщения во внешней системе |
| `ExternalMessageUpdatedDate` | дата последнего обновления внешнего статуса |
| `OutgoingStatus` | текущий статус отправки |
| `EditStatus` | результат редактирования сообщения |
| `DeletionStatus` | результат удаления сообщения |

Если connector возвращает успешный статус и внешний id, OCC обновляет
`ExternalMessageId` и передаёт status DTO в
`BPMSoftOCCChatMessageStatusRequestHandler`.

## Status callbacks

Статусы входят через `BPMSoftOCCChatRequestService`:

| Метод | Контракт |
| ----- | -------- |
| `SendMessageStatus(messageStatus)` | один bare status |
| `SendMessageStatuses(messageStatuses)` | batch bare statuses |
| `SendMessageStatusWrapped(requestBody)` | wrapped single status |
| `SendMessageStatusesWrapped(requestBody)` | wrapped batch statuses |

Handler ищет `BPMSoftOCCChatMessage` по `CrmMessageId` или
`ExternalMessageId`, обновляет `OutgoingStatus`, затем отправляет в client UI
сообщение:

```text
header = "MessageBatchStatusUpdate"
channel = "BPMSoftOCCChat"
```

Для Viber есть дополнительная логика: при статусе "прочитано" handler может
обновлять несколько сообщений в рамках чата.

## Edit/Delete callbacks

Edit/delete-операции не являются обычным входящим сообщением. Они идут через
специализированные обработчики:

| Handler | Назначение |
| ------- | ---------- |
| `BPMSoftOCCChatMessageEditMessageRequestHandler` | подтверждение или rollback текста |
| `BPMSoftOCCChatMessageDeleteMessageRequestHandler` | подтверждение или ошибка удаления |

При успехе/ошибке UI получает websocket payload:

```text
header = "MessageEditMessageInfo"
channel = "BPMSoftOCCChat"
```

## Неотправленные сообщения

Если исходящий вызов connector не дал успешный результат, `Api.SendMessage`
может создать запись `BPMSoftOCCNotSendMessage`.

Эта сущность нужна как диагностический и восстановительный след:

- исходный `BPMSoftOCCChatMessage` остаётся в CRM;
- ошибка не теряется в одном HTTP response;
- downstream job/ручная диагностика могут найти сообщения, требующие повтора.

## Типовые точки отказа

| Симптом | Где смотреть |
| ------- | ------------ |
| Сообщение создано, но нет внешнего id | `BPMSoftOCCApi.SendMessage`, connector response |
| Статус не меняется в UI | `BPMSoftOCCChatMessageStatusRequestHandler`, msg channel `BPMSoftOCCChat` |
| Статус приходит, но сообщение не находится | `CrmMessageId`, `ExternalMessageId`, фильтры status handler |
| Edit/delete не откатывается | `BPMSoftOCCOriginalChatMessage`, edit/delete handler |
| В логах есть not sent | `BPMSoftOCCNotSendMessage`, connector URL/settings |
