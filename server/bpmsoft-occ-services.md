# WCF-сервисы OCC и Sender

<!-- Версия: 1.0 | Обновлено: 2026-04-23 | Платформа: BPMSoft 1.9 -->
<!-- Теги: OCC, Sender, WCF, сервисы, chat, channel, whatsapp, delivery -->

> Профильный справочник по сервисам пакетов `BPMSoftOCC`, `BPMSoftOCCWAMfmsJson`, `BPMSoftSender`. Базовые принципы WCF/REST см. в [services.md](services.md).

## Обзор

OCC-контур активно использует WCF-сервисы как серверную точку входа для:

- UI операторов и страниц каналов;
- настройки connector и каналов;
- transfer/routing сценариев;
- получения и обработки входящих запросов из внешних мессенджеров;
- запуска и остановки рассылок Sender.

Базовый URL такой же, как и у остальных конфигурационных сервисов:

```text
/0/rest/{ServiceName}/{MethodName}
```

## `BPMSoftOCCChatService`

Основной сервис операторского UI. Делегирует большую часть логики в `ApiService` / `ApiServiceV2`.

Ключевые методы:

| Метод | Назначение |
| ----- | ----- |
| `ChangeStatus(userId, status, afkMode)` | Изменение статуса оператора |
| `PushChat(chatId, isStart)` | Запуск/продолжение обработки чата |
| `ProcessChat(chatId, isGroupChat)` | Передача чата в processing pipeline |
| `GetMessages()` | Инициация long-polling доставки сообщений при включённом feature flag |
| `UpdateClient(cid)` | Привязка OCC-клиента к контакту текущего пользователя |
| `UpdateUserInfo(message, chatId)` | Сохранение metadata посетителя чата |
| `GetMessageCount()` | Получение количества чатов/сообщений для UI |
| `CreateSiteChannel(channelId, domain)` | Создание site-channel экземпляра |
| `CloseChat(chatId, category, closeChatDescription, isGroupChat)` | Закрытие чата |
| `TransferChat(internalChatId, userId, isGroupChat)` | Передача чата другому оператору |
| `GetMyChats(chatId)` | Выборка чатов оператора |
| `GetMessagesByChatId(chatId, ...)` | История сообщений чата |
| `SendMessage(chatId, clientId, messageId, text, type, ...)` | Отправка сообщения |
| `SaveDraft(...)`, `GetDraftByChatId(...)` | Работа с draft-сообщениями |

Особенности:

- сервис наследуется от `BaseService`, поэтому работает через `UserConnection`;
- часть методов ориентирована на операторский интерфейс, часть - на embedded/site chat;
- `GetMessages()` зависит от feature flag `BPMSoftOCCMeesageLongPooling`.

Опорный файл: `Autogenerated/Src/BPMSoftOCCService.BPMSoftOCC.cs`.

## `BPMSoftOCCChatConfigService`

Сервис конфигурации connector:

| Метод | Назначение |
| ----- | ----- |
| `SendConfig(configId, host, closeChat, saveAfkChats, saveAfkChatTimeout, closeChatTimeout, switchOn)` | Отправка OCC-конфигурации во внешний connector |
| `ChangeBPMSoftInaccessibleTextInConnector()` | Обновление текстов/служебной конфигурации в connector |

Используется на этапе настройки OCC и администрирования конфигурации чатов.

## `BPMSoftOCCChatRequestService`

Ключевой входной слой для callback'ов и webhook payload от внешних каналов.

| Метод | Назначение |
| ----- | ----- |
| `SendBPMSoftOCCChatRequestCollection(requestBody)` | Приём request payload, запись в `BPMSoftOCCChatRequest`, вызов `RequestHandler.ProcessRequest(...)` |
| `SendTypingClientTextCollection(requestBody)` | Событие “клиент печатает” с доставкой в OCC UI |
| `SendMessageStatuses(messageStatuses)` | Batch-обработка статусов сообщений |
| `SendMessageStatus(messageStatus)` | Обработка одного статуса сообщения |
| `SendMessageStatusWrapped(requestBody)` | Wrapped-вариант single status |
| `SendMessageStatusesWrapped(requestBody)` | Wrapped-вариант batch statuses |

Особенности:

- OCC сохраняет raw request в `BPMSoftOCCChatRequest`, после чего передаёт его в request pipeline;
- поддерживаются и `Bare`, и `Wrapped` варианты контрактов;
- этот сервис важно учитывать при анализе интеграции с connector.

Опорный файл: `Autogenerated/Src/BPMSoftOCCChatRequestService.BPMSoftOCC.cs`.

## `BPMSoftOCCAddChannelService`

Универсальный сервис регистрации каналов.

Поддерживаемые сценарии:

- `AddViberChannel`
- `AddWeChatChannel`
- `AddVKChannel`
- `AddVKWallChannel`
- `AddTelegramChannel`
- `AddOKChannel`
- `AddFBChannel`
- `AddFBMChannel`
- `AddInstagramChannel`
- `AddAPIChannel`
- `AddBPMSoftOCCChannel`
- `AddSkypeChannel`
- `AddTeamsChannel`
- `AddWorkplaceChannel`
- `AddWhatsAppChannel`
- `AddSFBChannel`
- `AddTwitterChannel`
- `AddLineChannel`
- `AddDefaultChannel`
- `AddMaxChannel`

Отдельные служебные методы:

| Метод | Назначение |
| ----- | ----- |
| `GetConnectorUrl()` | Получение connector host/url из настроек |
| `GetBPMSoftOCCChannel(domain, channelId)` | Выборка/поиск channel-конфигурации |

Общие принципы работы:

- сервис собирает JSON payload в зависимости от `ChannelCode`;
- использует `BPMSoftOCCOperatorAppId` как часть интеграционного контекста;
- отправляет JSON во внешний connector API;
- при успехе создаёт запись в `BPMSoftOCCChannel`;
- в режиме `updateMode` может не создавать новую запись, а только перевалидировать внешний webhook/config.

Опорный файл: `Autogenerated/Src/BPMSoftOCCAddChannelService.BPMSoftOCC.cs`.

## `BPMSoftOCCSetupService`

Сервис мастера настройки OCC.

| Метод | Назначение |
| ----- | ----- |
| `SetupStageOne(login, password, token, bpmUrl)` | Первичная настройка логина/пароля и токена интеграции |
| `SetupPreviousStage()` | Возврат на предыдущий шаг мастера |
| `SetupNextStage()` | Переход к следующему шагу мастера |
| `SetupCloseAndTransfer(isClosed, isTransfer, closeTimeout, transferTimeout)` | Конфигурация close/AFK transfer поведения |

Особенности:

- использует `Scope = "bpmsoftocc"`;
- работает с настройками `CurrentSettingStep`, `AreConfiguredChats`, `BPMSoftOCCOperatorAppId`, `BPMSoftOCCOperatorToken`, `BPMSoftOCCTokenService`;
- обновляет сущность `BPMSoftOCCChatConfiguration`.

Опорный файл: `Autogenerated/Src/BPMSoftOCCSetupService.BPMSoftOCC.cs`.

## `BPMSoftOCCTransferService`

Сервис transfer-сценариев между операторскими группами.

Основной метод:

| Метод | Назначение |
| ----- | ----- |
| `StartTransferToGroupProcess(GroupId, ChatId)` | Выбор нового оператора по группе/очереди и перепривязка чата |

Внутри сервиса:

- определяется `ChannelId` чата;
- выбирается позиция оператора в группе;
- рассчитывается подходящий оператор с учётом `QueuePosition`, активности и текущей нагрузки;
- вызывается обновление `OperatorId` у `BPMSoftOCCChat`.

Опорный файл: `Autogenerated/Src/BPMSoftOCCTransferService.BPMSoftOCC.cs`.

## `BPMSoftOCCAddMfmsJsonService`

Специализированный сервис пакета `BPMSoftOCCWAMfmsJson` для WhatsApp.

| Метод | Назначение |
| ----- | ----- |
| `GetConnectorUrl()` | Получение базового URL connector host |
| `AddWhatsAppMfmsJsonChannel(...)` | Создание/обновление канала в режиме `wa/edna` |
| `AddObsoleteWhatsAppMfmsJsonChannel(...)` | Legacy-вариант `wa/mfms/json` |

Особенности:

- сервис создаёт webhook URL на стороне BPMSoft;
- отправляет JSON во внешний endpoint `/api/v1/webhook`;
- записывает JSON payload в `BPMSoftOCCChannel.Token`;
- поддерживает `updateMode`, когда запись канала не создаётся повторно.

Подробности по этому сценарию см. в [occ-whatsapp-mfms-json.md](occ-whatsapp-mfms-json.md).

## `BSDeliveryService`

Базовый сервис жизненного цикла рассылки в `BPMSoftSender`.

| Метод | Назначение |
| ----- | ----- |
| `StartDelivery(deliveryId)` | Запуск рассылки, подготовка sender job'ов и статусов |
| `StopDelivery(deliveryId)` | Остановка рассылки и перевод в finished |
| `ResetRecipientStatus(deliveryId)` | Сброс статусов получателей |

Особенности:

- взаимодействует с `BPMSoftOCCDelivery` и `BSDeliveryRecipient`;
- инициирует `BPMSoftSenderDeliverySchedulerJob` и `BPMSoftSenderStatusJob`;
- использует `BSDeliveryStrategy` как основной движок доставки.

Опорный файл: `Autogenerated/Src/BSDeliverySource.BPMSoftSender.cs`.

## `BSSchedulerService`

Сервис планирования запуска рассылок Sender.

| Метод | Назначение |
| ----- | ----- |
| `AddProcessToScheduleCronExp(processName, deliveryId, startDateJS, endDateJS, cronExpression)` | Планирование процесса по cron с окном Start/End |
| `AddProcessToScheduleSingleRun(processName, deliveryId, executionDateTimeJS)` | Одноразовый запуск в указанную дату |
| `RemoveProcessFromSchedule(processName, deliveryId)` | Удаление job и перевод статусов рассылки |
| `SetRunningDeliveryStatus(deliveryId)` | Перевод получателей в running и запуск sender background jobs |

Особенности:

- сервис планирует именно process job, а не прямую отправку сообщения;
- использует `DeliveryId` как главный ключ бизнес-сценария;
- при удалении расписания дополнительно обновляет `BPMSoftOCCDelivery` и `BSDeliveryRecipient`.

Подробности по Quartz см. в [scheduler-quartz.md](scheduler-quartz.md).

## `RecipientGroupService`

Сервис работы с группами получателей рассылки.

| Метод | Назначение |
| ----- | ----- |
| `AddRecipientGroup(currentDeliveryId, contactFolders)` | Создание связей delivery ↔ contact folder |
| `DeleteRecipientByDeliveryId(deliveryId)` | Удаление автоматически добавленных получателей рассылки |

Используется UI рассылки для заполнения и очистки `BSDeliveryRecipientGroup` / `BSDeliveryRecipient`.

Опорный файл: `Autogenerated/Src/RecipientGroupService.BPMSoftSender.cs`.

## Клиентские вызовы

Типовые OCC/Sender страницы вызывают эти сервисы через `ServiceHelper`:

```javascript
BPMSoft.ServiceHelper.callService(
    "BPMSoftOCCAddMfmsJsonService",
    "AddWhatsAppMfmsJsonChannel",
    callback,
    data,
    this
);
```

Примеры страниц:

- `Autogenerated/Src/BPMSoftOCCChannelPage.BPMSoftOCCWAMfmsJson.js`
- `Autogenerated/Src/BPMSoftOCCWhatsAppChannelView.BPMSoftOCCWAMfmsJson.js`
- `Autogenerated/Src/BPMSoftOCCDelivery1Page.BPMSoftSender.js`
