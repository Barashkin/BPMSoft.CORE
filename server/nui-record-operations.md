# NUI Record Operations

<!-- Версия: 1.0 | Обновлено: 2026-04-27 | Платформа: BPMSoft 1.9 -->
<!-- Теги: NUI, record operations, multi delete, grid, AppScheduler -->

> Серверный контур операций над записями из NUI grid/section: синхронное и
> асинхронное удаление, multi-link операции, проверка прав, очередь ошибок.

## Основные сценарии

| Сценарий | Точка входа |
| -------- | ----------- |
| Удалить выбранные записи | `GridUtilitiesService.DeleteRecords(...)` |
| Удалить выбранные записи асинхронно | `GridUtilitiesService.DeleteRecordsAsync(...)` |
| Выполнить операцию по фильтру | `filtersConfig` + `GetPrimaryColumnValuesFromFilters(...)` |
| Массово связать записи | `MultiOperationService.MultiLinkEntity(...)` |
| Сохранить ошибки delete | `MultiDeleteQueue` + `MultiDeleteDenyReason` |

## GridUtilitiesService

Опорный файл: `Autogenerated/Src/GridUtilitiesService.NUI.cs`.

`DeleteRecords(...)`:

1. Строит ESQ по `rootSchema`.
2. Фильтрует по primary column values.
3. Проверяет locked records через `CollisionLockedRecordChecker`.
4. Вызывает `entity.Delete()` для каждой записи.
5. Возвращает JSON с `Success`, `DeletedItems`, flags ошибок и сообщением.

`DeleteRecordsAsync(...)`:

1. Собирает параметры операции: `RecordIds`, `SchemaName`, `OperationKey`.
2. Если передан `filtersConfig`, сначала разворачивает фильтр в ids.
3. Для малого набора запускает `MultiDeleteExecutor` синхронно.
4. Для большого набора создаёт class job через `AppScheduler.TriggerJob`.

Размер порога берётся из `DefaultNumberItemsReturned`.

## Worker / executor model

Bulk pipeline разделён на несколько ролей:

| Класс / интерфейс | Роль |
| ----------------- | ---- |
| `IRecordOperationExecutor` | контракт операции над одной записью |
| `BaseRecordExecutor` | базовая проверка record rights и выполнение операции |
| `IRecordsOperationWorker` | контракт обработки набора записей |
| `BaseRecordsOperationWorker` | ESQ, distinct, цикл обработки, event `RecordProcessed` |
| `IRecordProcessedHandler` | обработчик результата обработки записи |
| `MultiDeleteOperationAgent` | запись прогресса/ошибок в очередь multi delete |

`BaseRecordExecutor` проверяет права через `DBSecurityEngine` перед
операцией. Это отличает security failure от domain/db failure.

## MultiDeleteQueue

`MultiDeleteQueue` хранит ошибки и состояние массового удаления. Связанный
lookup `MultiDeleteDenyReason` классифицирует причину отказа.

`MultiDeleteOperationAgent` использует:

- `OperationKey` для группировки операции;
- `SchemaName` для схемы удаляемых записей;
- `DenyReason` для объяснения отказа;
- lifetime-константы для очистки queue/error records.

Логгер для этого контура: `MultiDelete`.

## MultiOperationService

Опорный файл: `Autogenerated/Src/MultiOperationService.NUI.cs`.

`MultiLinkEntity(...)` принимает:

- `entityName`;
- `recordsId`;
- `parameters`;
- `filtersConfig`.

Если `filtersConfig` передан, ids вычисляются сервером через
`GridUtilitiesServiceHandler.GetPrimaryColumnValuesFromFilters(...)`.
После этого сервис создаёт `MultiLinkEntity` strategy и передаёт её в
`MultiOperationProvider`.

## Граница с Security

Security dive описывает права как модель: schema rights, record rights,
operation rights. Этот документ описывает, где NUI применяет эти проверки в
операциях grid/record.

Практическое правило: client-side disabled button не заменяет server-side
проверку. Массовая операция должна проверять каждую запись.
