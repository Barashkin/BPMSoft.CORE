## Общая модель

Сервисы конфигурации в BPMSoft обычно выглядят как WCF-классы с REST/JSON endpoint'ами.

Базовый URL:

```text
/0/rest/{ServiceName}/{MethodName}
```

Минимальные элементы:

- `[ServiceContract]` на классе;
- `[OperationContract]` на методе;
- `[WebInvoke]` для REST/JSON маршрута;
- `UserConnection` из `BaseService` или session;
- response DTO или простое значение.

## Типовые серверные варианты

| Вариант | Когда используется | Пример |
| ----- | ----- | ----- |
| `BaseService` + JSON DTO | обычный конфигурационный сервис | `CompletenessService.Completeness.cs` |
| `BaseService` + `IReadOnlySessionState` | чтение без записи в session | `BPMSoftOCCChatRequestService.BPMSoftOCC.cs` |
| `DefaultServiceRoute` + `SspServiceRoute` | доступ из основного приложения и SSP | `FileApiService.NUI.cs` |

## Когда нужен сервис

Сервис нужен, если клиентский код должен:

- вызвать серверную бизнес-логику;
- получить данные, которые нельзя безопасно собрать на клиенте;
- выполнить действие с правами пользователя;
- обратиться к внешней системе через сервер;
- обработать webhook/callback от внешнего connector.

## Когда сервис не нужен

Не стоит создавать сервис, если:

- достаточно client ESQ;
- действие уже покрыто existing service/helper;
- нужна только массовая операция на данных внутри server-side job/event listener;
- endpoint дублирует уже существующий REST/WCF метод.
