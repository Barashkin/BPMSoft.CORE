# SchedulerJobService

<!-- Версия: 1.0 | Обновлено: 2026-04-27 | Платформа: BPMSoft 1.9 -->
<!-- Теги: Quartz, SchedulerJobService, WCF, REST, client-server -->

> Узкий документ по WCF-сервису `SchedulerJobService`: методы, реальные ограничения, client-side вызовы и named scheduler support.

## Обзор

`SchedulerJobService` - server-side фасад над `AppScheduler` для клиентского кода.

URL сервиса:

```text
/0/rest/SchedulerJobService/{Method}
```

Namespace:

- `BPMSoft.Configuration.SchedulerJobService`

## Методы сервиса

| Метод | Назначение |
| ----- | ----- |
| `CreateSyncJob` | создать периодическую process-based sync job |
| `CreateSyncJobWithResponse` | то же, но с `ConfigurationServiceResponse` |
| `CreateImmediateSyncJob` | немедленный запуск процесса |
| `CreateSyncJobByPeriodicity` | создать trigger по `PeriodicitySettings` |
| `CheckIfJobExist` | проверить наличие задачи |

## Что реально делает `CreateSyncJob`

Серверный код:

1. получает scheduler через `AppScheduler.GetSchedulerOrDefault(schedulerName)`;
1. всегда вызывает `RemoveJob(jobName, syncJobGroupName, scheduler)`;
1. если `periodInMinutes < 0` или `recreate == false`, просто завершает работу;
1. иначе вызывает `ScheduleMinutelyProcessJob(...)`.

Следствие:

- метод работает как “пересоздай периодическую задачу, если это разрешено”.

## Особенность `CreateSyncJobWithResponse`

Если `periodInMinutes == 0`, сервис не создаёт периодическую задачу, а переключается на:

- `CreateImmediateSyncJob(...)`

Это особенно важно для UI-сценариев “запусти сейчас”, но через тот же service endpoint.

## `CreateSyncJobByPeriodicity`

Этот метод не принимает cron напрямую.
Он:

1. получает `PeriodicitySettingsUtilities(UserConnection, id)`;
1. вызывает `CreateTrigger(...)`;
1. тем самым переводит запись `PeriodicitySettings` в Quartz trigger.

Подробности и ограничения см. в [quartz-periodicity-settings.md](quartz-periodicity-settings.md).

## Named scheduler support

Во всех основных public-методах есть optional параметр:

- `schedulerName = null`

Это позволяет вызывать сервис не только для default scheduler, но и для отдельных scheduler instance.

Реальный пример:

- LDAP UI вызывает `CreateSyncJobWithResponse` с `schedulerName: "LdapQuartzScheduler"`

## Client-side вызовы

### Google integration settings

В `GoogleIntegrationSettingsModule.NUI.js` и `GoogleIntegrationSettingsPageV2.NUI.js` встречаются вызовы:

- `CheckIfJobExist`
- `CreateSyncJob`

Типовой payload:

```json
{
  "JobName": "SomeJob_<CurrentUserId>",
  "SyncJobGroupName": "SomeGroup",
  "SyncProcessName": "SomeProcess",
  "periodInMinutes": 30,
  "recreate": true
}
```

### LDAP UI

В `SysAdminUnitSectionV2.LDAP.js` используется:

- `serviceName: "SchedulerJobService"`
- `methodName: "CreateSyncJobWithResponse"`
- `schedulerName: "LdapQuartzScheduler"`
- `periodInMinutes: 0`

То есть UI использует сервис и для immediate запуска процесса в именованном scheduler.

## Что важно знать клиентскому коду

### 1. `CheckIfJobExist` возвращает не “raw bool из responseText”

В старом клиентском коде результат читается из поля:

- `CheckIfJobExistResult`

### 2. `CreateSyncJobWithResponse` возвращает wrapper-объект

Клиент читает:

- `CreateSyncJobWithResponseResult`

Внутри уже лежит `ConfigurationServiceResponse`.

### 3. Поведение при `recreate = false`

Сервис всё равно удаляет текущую задачу до проверки флага, а новую уже не создаёт.

Это важно: `recreate = false` фактически может означать “снять job и не восстанавливать её”.

## Где сервис полезен

`SchedulerJobService` стоит использовать, когда:

- расписанием управляет UI;
- нужно дать клиенту простой REST/WCF endpoint;
- требуется optional named scheduler;
- запускается process-based job, а не полноценный `ClassJob`.

## Когда сервиса недостаточно

Нужен другой слой, если:

- задача должна быть `ClassJob`;
- требуется сложная recovery-логика;
- нужно управлять misfire, trigger window или custom trigger-моделью;
- необходимо self-healing поведение на сервере.
