# OCC/Sender Troubleshooting

<!-- Версия: 1.1 | Обновлено: 2026-04-27 | Платформа: BPMSoft 1.9 -->
<!-- Теги: OCC, troubleshooting, logger, webhook, status, routing, sender -->

> Практический документ по диагностике OCC/Sender. Основан на реальных лог-сообщениях и recovery-паттернах из `Autogenerated/Src`, а не на общих советах.
> Для быстрого выбора между входящим pipeline, исходящим connector flow,
> routing, Sender и jobs см. [occ-omnichannel-overview.md](occ-omnichannel-overview.md).

## Обзор

Удобно делить проблемы OCC на 6 групп:

1. канал не создаётся;
1. webhook/request не обрабатывается;
1. сообщение не уходит или не получает статус;
1. routing не назначает оператора;
1. AFK/logout/auto-close работают не так, как ожидается;
1. UI не получает websocket-обновления.

## Главные точки диагностики

### 1. `BPMSoftOCC.Logger`

По коду OCC логирует в `BPMSoftOCC.Logger.Log(...)` и `BPMSoftOCC.Logger.Trace(...)`.

Наиболее полезные маркеры:

| Маркер | Где возникает | Что означает |
| ----- | ----- | ----- |
| `EXCEPTION(ChatRequestInserted)` | request service / entity event | ошибка разбора или обработки request payload |
| `EXCEPTION(TypingClientTextInserted)` | typing callback | ошибка в typing-flow |
| `Unknown status code ...` | status handler | connector прислал неподдерживаемый `StatusCode` |
| `SendMessageException` | `BPMSoftOCCApi.SendMessage` | ошибка отправки исходящего сообщения |
| `SendMessageResponse` | `BPMSoftOCCApi.SendMessage` | сырой ответ connector при ошибке |
| `RoutingInfo` | `ChatRoutingStrategy` | нет операторов, нет чатов для маршрутизации, оператор не найден |
| `RoutingException` | `ChatRoutingStrategy` | ошибка внутри routing-алгоритма |
| `MutexException` | `RequestHandler` | сбой в синхронизации request по отправителю |
| `UpdateQueryStrategy` | status update | ошибка обновления статуса в `BPMSoftOCCChatMessage` |

### 2. Таблицы/сущности, которые смотреть первыми

| Сущность | Когда проверять |
| ----- | ----- |
| `BPMSoftOCCChannel` | канал не создаётся или создан с пустыми credential fields |
| `BPMSoftOCCChatRequest` | webhook пришёл, но pipeline не завершился |
| `BPMSoftOCCChat` | чат не назначился оператору или неожиданно закрылся |
| `BPMSoftOCCChatMessage` | исходящее сообщение, статусы, edit/delete |
| `BPMSoftOCCNotSendMessage` | сообщение не отправилось |
| `BPMSoftOCCChatOperator` | история назначения операторов |
| `BPMSoftOCCAfkTransfer` | AFK transfer был зафиксирован |
| `BPMSoftOCCDelivery`, `BSDeliveryRecipient` | проблемы Sender-рассылок |

## Канал не создаётся

### Симптомы при создании канала

- channel view показывает ошибку после `Add*Channel`;
- webhook URL не формируется;
- запись в `BPMSoftOCCChannel` не создаётся;
- connector отвечает `BadRequest` или `Unauthorized`.

### Проверить для канала

1. SysSettings:
   - `BPMSoftOCCServices`
   - `BPMSoftOCCOperatoHost`
   - `BPMSoftOCCOperatorAppId`
1. конкретный `BPMSoftOCC*ChannelView*.js` и набор его полей;
1. соответствующий метод в `BPMSoftOCCAddChannelService`;
1. содержимое сохранённых полей `Token`, `InternalId`, `ClientId`, `ClientSecret`.

### Частые причины ошибки канала

- неверный connector host в `GetConnectorUrl()`;
- неправильный payload для выбранного `ChannelCode`;
- UI отправляет не те поля для конкретного view;
- credentials сохранены не в ту колонку `BPMSoftOCCChannel`.

### Отдельный случай: WhatsApp MFMS/Edna

Если проблема касается MFMS/Edna, сразу проверь:

- `BPMSoftOCCUseObsoleteWhatsAppApi`;
- JSON внутри `BPMSoftOCCChannel.Token`;
- различие endpoint `wa/edna` vs `wa/mfms/json`.

## Webhook/request пришёл, но чат не появился

### Симптомы в request pipeline

- внешний канал утверждает, что webhook доставлен;
- новый чат/сообщение не создаются;
- в UI нет `NewChat`;
- в `BPMSoftOCCChatRequest` остаются необработанные записи.

### Проверить для request pipeline

1. есть ли запись в `BPMSoftOCCChatRequest`;
1. заполнено ли `Request`;
1. остаётся ли `Processed = false`;
1. есть ли лог `EXCEPTION(ChatRequestInserted)`;
1. существует ли `ChannelId` из payload в `BPMSoftOCCChannel`;
1. корректен ли `Sender.InternalId`.

### Что означает `Processed = false`

Это главный сигнал, что request:

- не дошёл до конца в `RequestHandler`;
- либо упал раньше `AfterProcessRequestStrategy`;
- либо recovery job ещё не обработал его повторно.

### Куда смотреть дальше

- `BPMSoftOCCChatRequestService.SendBPMSoftOCCChatRequestCollection(...)`
- entity event `BPMSoftOCCChatRequestInserted`
- `RequestHandlingJob`
- `RequestHandler.ProcessRequest(...)`
- сводная карта jobs: [occ-jobs-map.md](occ-jobs-map.md)

## Typing event не доходит до UI

### Симптомы по typing event

- клиент печатает, но оператор не видит индикатор;
- других проблем с чатами может не быть.

### Проверить для typing event

1. вызывается ли `SendTypingClientTextCollection(...)`;
1. находит ли сервис ровно один открытый чат;
1. нет ли `EXCEPTION(TypingClientTextInserted)`;
1. приходит ли websocket header `ClientIsTyping` в `BPMSoftOCCChatSchema`.

### Типовая причина

Сервис специально возвращает `Ok`, если открытых чатов не один. Поэтому typing может “молча” не отобразиться при дублях/нескольких открытых чатах того же клиента.

## Сообщение не отправляется

### Симптомы по отправке сообщения

- оператор отправил сообщение, но статус завис;
- появляется error/not delivered иконка;
- в connector есть ошибка;
- в БД появляется запись о неотправленном сообщении.

### Проверить для исходящего сообщения

1. логи `SendMessageException` и `SendMessageResponse`;
1. запись сообщения в `BPMSoftOCCChatMessage`;
1. поле `OutgoingStatusId`;
1. наличие записи в `BPMSoftOCCNotSendMessage`.

Подробная карта исходящего flow: [occ-outgoing-connector.md](occ-outgoing-connector.md).

### Что означает `BPMSoftOCCNotSendMessage`

Сущность `BPMSoftOCCNotSendMessage` хранит ссылку `MessageId -> BPMSoftOCCChatMessage` и используется как буфер/маркер неотправленных сообщений.

Если запись там появилась, проблема уже была зафиксирована на этапе отправки.

## Статусы доставки не обновляются

### Симптомы по delivery status

- сообщение ушло, но UI остаётся в `created`;
- read/delivered не приходят;
- отдельные каналы дают непредсказуемые статусы.

### Проверить для status callback

1. приходит ли callback в `SendMessageStatus(...)` или `SendMessageStatuses(...)`;
1. есть ли лог `Unknown status code ...`;
1. заполнен ли `ExternalMessageId`;
1. обновляется ли `ExternalMessageUpdatedDate`;
1. приходит ли websocket `MessageBatchStatusUpdate`.

### Частые причины проблем со статусами

- connector прислал status code, которого нет в lookup `BPMSoftOCCChatMessageOutgoingStatus`;
- `CrmMessageId`/`ExternalMessageId` не соответствует существующему сообщению;
- новый статус меньше или равен текущему, поэтому handler его игнорирует;
- фронт не получил websocket после обновления.

### Особый случай Viber

Для Viber `ExternalIdIsNotNullStrategy` специально обновляет несколько сообщений чата при статусе `Read`. Поэтому отсутствие массового перехода в `Read` обычно означает проблему именно в external-id match.

## Edit/Delete callback не сходится

### Симптомы по edit/delete callback

- сообщение “редактируется бесконечно”;
- текст откатывается назад;
- delete/edit иконки не совпадают с фактическим состоянием.

### Проверить для edit/delete

1. `EditStatusId` и `DeletionStatusId` в `BPMSoftOCCChatMessage`;
1. наличие записи в `BPMSoftOCCOriginalChatMessage`;
1. приходит ли websocket `MessageEditMessageInfo`;
1. совпадает ли `StatusCode` с ожидаемыми `200`/`300`.

### Интерпретация

- если `BPMSoftOCCOriginalChatMessage` не удаляется, edit-flow не завершён;
- если пришёл `300`, обработчик делает rollback текста;
- если delete/edit callback успешен, UI должен получить `MessageEditMessageInfo`.

## Чат не назначается оператору

### Симптомы по routing

- чат есть, но `OperatorId` пустой;
- чат зависает без назначения;
- routing логирует `No operators set` или `Chat operator ... not found`.

### Проверить для routing

1. есть ли у чата `ChannelId` и не закрыт ли он;
1. есть ли активные операторы `Status = 2`, `Active = true` для этого канала;
1. хватает ли оператору `Weight` под вес канала;
1. не блокирует ли routing widget/bot-ограничение;
1. есть ли записи в `BPMSoftOCCChatOperator`.

### Частые причины сбоев routing

- операторы не привязаны к каналу;
- оператор offline или inactive;
- суммарный вес открытых чатов уже достиг `MaxWeight`;
- group chat исключил bot operators;
- чат уже помечен synchronizer'ом как обрабатываемый.

## AFK transfer работает неожиданно

### Симптомы по AFK transfer

- чат перекинулся другому оператору “сам”;
- transfer случился позже или раньше ожидаемого;
- пользователь ожидает auto-close, а получает transfer.

### Проверить для AFK transfer

1. запускался ли `SaveAFKChatJob`;
1. время последнего `StartDate` в `BPMSoftOCCChatOperator`;
1. последнее сообщение в чате и `IsOperatorLastMessage`;
1. создалась ли запись в `BPMSoftOCCAfkTransfer`;
1. исполнился ли процесс `BPMSoftOCCLogOperatorChange`.

### Ключевое различие

- `SaveAFKChatJob` пытается перепривязать чат;
- `ScheduleOperatorLogoutJob.AutoCloseChat(...)` просто закрывает чат.

Если чат закрылся вместо transfer, проблема, скорее всего, в logout/auto-close параметрах, а не в AFK-routing.

## Оператор “вечно online” или неожиданно offline

### Проверить для auto-offline

1. job `ScheduleOperatorLogoutJob`;
1. таблицу `SysUserSession`;
1. текущее `Status` в `BPMSoftOCCOperatorUnit`.

`LogoutHandler(...)` переводит offline операторов без активной сессии. Если статус сбрасывается неожиданно, сначала ищи сессионную проблему, а не UI-баг.

## Чат неожиданно закрывается

### Проверить для auto-close

1. параметр `closeChat` для `ScheduleOperatorLogoutJob`;
1. `closeChatTimeout`;
1. дату последнего сообщения в `BPMSoftOCCChatMessage`;
1. кто менял `Closed = true` у `BPMSoftOCCChat`.

Типовая причина: auto-close по timeout, а не пользовательское действие.

## UI не получает обновления

### Симптомы по websocket/UI

- в БД изменения есть, но правый чат не обновляется;
- статусы или typing не появляются;
- счётчик в communication panel не меняется.

### Проверить для websocket/UI

1. дошло ли websocket-сообщение в канал `BPMSoftOCCChat`;
1. правильный ли `header` у сообщения:
   - `NewChat`
   - `NewMessage`
   - `ClientIsTyping`
   - `MessageBatchStatusUpdate`
   - `MessageEditMessageInfo`
1. фильтр `to` в `BPMSoftOCCChatSchema.onWebSocketMessage(...)`;
1. соответствующий `sandbox.publish(...)`.

### Частая ловушка

Сообщение может успешно уйти в websocket, но UI его отбросит из-за `msg.to` или из-за того, что ожидается другой `header`.

## Sender: рассылка не стартует или зависает

### Проверить для Sender

1. `BPMSoftOCCDelivery` и его статус;
1. `BSDeliveryRecipient`;
1. вызовы `BSDeliveryService` и `BSSchedulerService`;
1. job'ы `BPMSoftSenderDeliverySchedulerJob`, `BPMSoftSenderStatusJob`, `BSChatRoutingJob`;
1. OCC channel configuration, если рассылка использует тот же канал доставки.

### Маркеры

- `BSDelEvent`
- `Exception saved Recipient`

Если у рассылки не обновляются recipient statuses, ищи проблему не только в Sender, но и в OCC delivery/status pipeline.

## Быстрая triage-таблица

| Симптом | Проверить первым |
| ----- | ----- |
| Канал не создаётся | `BPMSoftOCCServices`, `BPMSoftOCCOperatoHost`, channel view, `Add*Channel` |
| Нет нового чата | `BPMSoftOCCChatRequest`, `Processed`, `EXCEPTION(ChatRequestInserted)` |
| Нет typing | `SendTypingClientTextCollection`, открытый чат, websocket `ClientIsTyping` |
| Сообщение не ушло | `SendMessageException`, `BPMSoftOCCNotSendMessage`, `OutgoingStatusId` |
| Нет delivery status | `Unknown status code`, `ExternalMessageId`, `MessageBatchStatusUpdate` |
| Чат без оператора | `RoutingInfo`, operator-channel mapping, `Weight`, `Status = 2` |
| Чат сам перевёлся | `SaveAFKChatJob`, `BPMSoftOCCAfkTransfer` |
| Чат сам закрылся | `ScheduleOperatorLogoutJob`, `closeChatTimeout` |
| UI молчит | websocket `header`, `msg.to`, `sandbox.publish` |
