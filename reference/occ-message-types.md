# Типы сообщений и статусы OCC

<!-- Версия: 1.0 | Обновлено: 2026-04-27 | Платформа: BPMSoft 1.9 -->
<!-- Теги: OCC, message, status, edit, delete, file, image, location, buttons, carousel -->

> Справочник по типам сообщений OCC: базовая сущность `BPMSoftOCCChatMessage`, enum `MessageType`, rich-content сценарии и lookup'и статусов исходящих, отредактированных и удалённых сообщений.

## Обзор

В OCC сообщение почти всегда строится вокруг `BPMSoftOCCChatMessage`, а дополнительные типы контента и статусы навешиваются поверх него:

- `MessageType` определяет семантику сообщения;
- отдельные поля и служебные сущности хранят rich-content;
- lookup'и статусов отвечают за delivery/edit/delete состояние;
- часть типов представлена отдельными `ProcessUserTask`.

## Базовая сущность `BPMSoftOCCChatMessage`

Схема находится в `Autogenerated/Src/BPMSoftOCCChatMessageSchema.BPMSoftOCC.cs`.

Ключевые поля:

| Поле | Назначение |
| ----- | ----- |
| `Operator` | Оператор-автор |
| `Client` | Клиент чата |
| `Text` | Текст сообщения |
| `SendByOperator` | Признак исходящего сообщения |
| `Chat` | Приватный чат |
| `GroupChat` | Групповой чат |
| `MessageType` | Числовой тип сообщения |
| `Buttons` | JSON/строковое представление кнопок |
| `ConnectorUrl` | URL коннектора / вложения |
| `ImageUrl` | URL изображения |
| `IsDraft` | Draft-сообщение |
| `QuotationInfo`, `QuotationTime` | Цитирование/quotation |
| `InternalMessengerId` | Внутренний id мессенджера |
| `ParentPostData` | Данные родительского сообщения/поста |
| `Status` | Общий статус сообщения |
| `OutgoingStatus` | Lookup статуса исходящей доставки |
| `ExternalMessageUpdatedDate` | Время обновления внешнего статуса |
| `ExternalMessageId` | Внешний id сообщения |
| `EditStatus` | Lookup статуса редактирования |
| `DeletionStatus` | Lookup статуса удаления |

## Enum `MessageType`

В request pipeline (`BPMSoftOCCStrategy.BPMSoftOCC.cs`) используется enum:

| Значение | Смысл |
| ----- | ----- |
| `Text` | Обычный текст |
| `Image` | Изображение |
| `File` | Файл |
| `Audio` | Аудио |
| `Location` | Геолокация |
| `Contact` | Контакт |
| `Buttons` | Сообщение с кнопками |
| `Hero` | Hero/card сообщение |
| `List` | List/carousel-представление |
| `CloseChat` | Системное закрытие чата |
| `Initial` | Начальное/system сообщение |
| `OperatorInfo` | Информация об операторе |
| `Quotation` | Сообщение с цитированием |
| `DeleteMessage` | Удаление сообщения |
| `EditText` | Редактирование текста |

Практически это enum для транспорта и pipeline, а не полный список UI-компонентов.

## Основные группы сообщений

### Текстовые

Базовый тип - `Text`.

Хранение:

- `Text`
- `MessageType = Text`
- связь с `Chat` или `GroupChat`

### Медиа

Для медиа в базовой сущности используются общие поля, а в процессах есть специальные user tasks:

| Тип | Что используется |
| ----- | ----- |
| `Image` | `ImageUrl` + `BPMSoftOCCImageMessage` |
| `File` | `ConnectorUrl`/chat file + `BPMSoftOCCFileMessage` |
| `Audio` | отдельный enum type, обычно без отдельного deep model в текущем наборе файлов |

#### `BPMSoftOCCImageMessage`

Process user task принимает:

- `ChatId`
- `ClientId`
- `OperatorId`
- `Image`

#### `BPMSoftOCCFileMessage`

Process user task принимает:

- `ChatId`
- `ClientId`
- `OperatorId`
- `File`

### Геолокация

`BPMSoftOCCLocationMessage` принимает:

- `ChatId`
- `ClientId`
- `OperatorId`
- `Latitude`
- `Longitude`
- `Text`

То есть location message в OCC - это не только координаты, но и сопроводительный текст.

### Кнопки

Для интерактивного сообщения с кнопками используются:

- `MessageType = Buttons`;
- поле `Buttons` в `BPMSoftOCCChatMessage`;
- process user task `BPMSoftOCCButtonMessage`.

`BPMSoftOCCButtonMessage` поддерживает:

- `Text`
- до 10 кнопок (`Button1` ... `Button10`)
- `Awaitable`
- `WaitAnswer`
- `Answer`

На исполнении task вызывает `ButtonMessageElement`, сохраняет состояние по `ChatId` и позже снимает его в `CompleteExecuting(...)`.

### Carousel / hero / list

В коде rich-card сценарий представлен `BPMSoftOCCCaruselMessage`.

Он формирует JSON `Config` вида:

- `Title`
- `List`
- внутри `List` до 5 элементов с `Title`, `Image`, `Button`

С точки зрения enum такие сценарии ближе всего к `Hero`/`List`, но в реализации используются отдельный task и конфигурационный JSON.

### Цитирование и post-linked сообщения

Для quoted/post сообщений в `BPMSoftOCCChatMessage` есть поля:

- `QuotationInfo`
- `QuotationTime`
- `ParentPostData`

Они позволяют не заводить отдельный тип-сущность, а хранить дополнительные метаданные рядом с базовым сообщением.

## Исходящие статусы сообщений

Lookup `BPMSoftOCCChatMessageOutgoingStatus` хранит маппинг между бизнес-статусом и числовым кодом `MessageStatusCode`.

Ключевые поля lookup:

| Поле | Назначение |
| ----- | ----- |
| `Name` | Отображаемое имя статуса |
| `MessageStatusCode` | Числовой код от connector |

### Как статусы применяются

`BPMSoftOCCChatMessageStatusRequestHandler`:

1. получает status callback;
1. загружает статус по `StatusCode`;
1. обновляет `OutgoingStatusId` у `BPMSoftOCCChatMessage`;
1. ставит `ExternalMessageUpdatedDate`;
1. отправляет websocket `MessageBatchStatusUpdate`.

### Наблюдаемые коды

По коду явно используются:

| Код | Смысл |
| ----- | ----- |
| `100` | `Created`, игнорируется при UI/update логике |
| `200` | `Sent` |
| `300` | `NotSent` |
| `600` | `Read` |

Важно: актуальный набор lookup-записей определяется данными БД, а не только кодом.

## Статусы редактирования

Lookup `BPMSoftOCCChatMessageEditStatus` отвечает за edit lifecycle.

В `BPMSoftOCCChatMessageEditMessageRequestHandler` явно используются состояния:

- `InProgress`
- `Edited`
- `EditError`

### Как работает edit callback

При успехе:

- удаляется запись из `BPMSoftOCCOriginalChatMessage`;
- в `BPMSoftOCCChatMessage` ставится `EditStatusId = Edited`;
- по websocket отправляется `MessageEditMessageInfo`.

При ошибке:

- текст откатывается из `BPMSoftOCCOriginalChatMessage`;
- ставится `EditStatusId = EditError`;
- отправляется websocket с rollback-содержимым.

`BPMSoftOCCOriginalChatMessage` здесь выступает как temporary storage оригинального текста до подтверждения редактирования.

## Статусы удаления

Lookup `BPMSoftOCCChatMessageDeletionStatus` отвечает за delete lifecycle.

В delete handler используются состояния:

- `Deleted`
- `DeletionError`

При callback:

- обновляется `DeletionStatusId`;
- обновляется `ModifiedOn`;
- операторский UI получает `MessageEditMessageInfo` с текущими `editStatus` и `deletionStatus`.

## Связанные служебные сущности

| Сущность | Назначение |
| ----- | ----- |
| `BPMSoftOCCOriginalChatMessage` | Хранение оригинального текста на время edit flow |
| `BPMSoftOCCNotSendMessage` | Буфер/фиксация неотправленных сообщений |
| `BPMSoftOCCMessageContentType` | Sender-side классификация content type |
| `BPMSoftOCCHotMessage`, `BPMSoftOCCHotMessageType` | Быстрые/преднастроенные сообщения |

## Как читать модель OCC-сообщений

Полезная практическая схема:

1. Базовая запись почти всегда живёт в `BPMSoftOCCChatMessage`.
1. `MessageType` говорит, как сообщение интерпретировать.
1. Rich-content может храниться в дополнительных полях (`Buttons`, `ImageUrl`, `ConnectorUrl`, `QuotationInfo`) или приходить через отдельный process task.
1. Delivery/edit/delete статусы живут в lookup-связях, а не в самом enum типа сообщения.

## Когда смотреть этот документ

Открывай его, если нужно:

- понять, какой `MessageType` приходит из pipeline;
- различить message type и delivery status;
- разобраться, где живут кнопки/carousel/location;
- отладить edit/delete callback;
- понять, почему сообщение отмечено как `NotSent`, `Edited` или `Deleted`.
